# TerminalLove — 최종 기획서 (구현 기준)
> *AI 에이전트 간 소개팅 프로토콜 (Agent Dating Protocol)*

> **한 줄 정의**: 개발 에이전트(Claude Code · Codex · Cursor) 안에서 작동하는, **오픈소스 에이전트 간 소개팅 프로토콜.**
> 데이팅 앱을 만드는 게 아니라, **Claude Code 시대의 소개팅 프로토콜**을 만든다.

| 항목 | 내용 |
|---|---|
| 문서 버전 | v1.0 (구현 착수 기준) |
| 확정 방향 | **오픈소스 코어 + 선택적 hosted layer** (완전 중앙 X, 완전 무정부 P2P X) |
| 네이밍 | **TerminalLove로 통일 (확정)** — 제품/프로토콜/레퍼런스 클라이언트 모두 `TerminalLove`. "Agent Dating Protocol"은 일반 카테고리 설명어로만 사용 |
| 스택 (확정) | **TypeScript + 공식 MCP SDK** / 암호 tweetnacl(Ed25519·X25519) |
| Phase 1 (확정) | **단일 머신 데모 풀 플로우** — init→profile→publish→scan→intro→open/coach |
| 태그라인 | **"Your agent knows you. The protocol only connects you."** / "당신을 분석하는 건 당신의 에이전트입니다. 프로토콜은 연결만 합니다." |
| 핵심 UX 문장 | **"개발하다가 터미널에서 소개팅한다."** |

---

## 0. 이 문서의 위치

원본 브레인스토밍에는 두 가지 버전이 섞여 있었다.

1. **초기안** — 중앙 서버가 매칭·라우팅·필터링을 모두 책임지는 "AI 붙인 데이팅 앱"
2. **최종안** — public key 신원 기반 **오픈소스 프로토콜 + 선택적 hosted layer**

본 기획서는 **최종안으로 확정**하고, 초기안에서 나온 보안/안전 논의(라우팅 위조 방지, 프롬프트 인젝션 방어, 신고/차단 등)는 hosted layer와 클라이언트 정책으로 흡수해 정리한다.

## 1. 제품 개요

### 1.1 무엇인가
사용자는 별도 데이팅 앱을 켜지 않는다. 평소 쓰던 개발 에이전트에 **TerminalLove MCP 서버를 연결**하면:

1. 로컬 에이전트가 사용자의 성향을 분석해 **성향 카드**를 만들고,
2. 사용자가 승인한 **공개 프로필(서명됨)**만 디렉토리에 게시되며,
3. 내 MCP가 디렉토리에서 프로필을 가져와 **로컬에서 매칭을 계산**하고,
4. **intro → accept → encrypted chat** 순으로 대화가 시작되며,
5. 대화 중 내 에이전트가 **답장을 코칭**한다.

```
$ claude
✓ tests passed
✓ build completed

🔔 New Match Found
   Name: Sofia
   Country: Spain
   Compatibility: 87%
   Stack: TypeScript, React, AI Tools

Type /dating intro to connect (1:1)
```

이 장면 자체가 가장 강한 바이럴 포인트다.

### 1.2 무엇이 신박한가
핵심은 "AI가 소개팅을 도와준다"가 아니라 **"내가 매일 쓰는 에이전트가 나를 가장 잘 알고, 그 에이전트가 인간관계까지 돕는다"**는 점이다.

기존 데이팅 앱: 사용자가 자기소개를 직접 쓴다 → 과장·빈약·천편일률.
ADP: **AI가 관찰해 프로필을 만든다** → 더 깊고 자연스럽다. 단, **분석은 로컬에서만** 하고 사용자가 승인한 요약만 공개한다.

## 2. 문제 정의

1. **프로필 신뢰도 부족** — 사람은 자기 객관화가 어렵고 소개글이 비슷해진다.
2. **매칭 후 대화 난이도** — 무엇을, 언제 보낼지, 상대가 관심 있는지 판단이 어렵다(특히 텍스트엔 익숙하나 플러팅엔 약한 개발자).
3. **글로벌 매칭의 언어/문화 장벽.**
4. **새 앱을 켜는 마찰** — 개발자는 이미 터미널/IDE/에이전트 안에서 하루를 보낸다.
5. **데이터 신뢰 문제** — "내 Claude Code 대화를 중앙 서버가 본다"는 거부감.

TerminalLove의 해법: **프로필은 AI가 만들고**, **대화는 AI가 코칭하며**, **이미 쓰는 에이전트 안에서** 만나고, **데이터는 로컬에 남는다.**

## 3. 핵심 컨셉 & 차별점

```
사람 ↔ 나의 AI 에이전트
            ↕
   매칭/디스커버리/라우팅 (오픈 프로토콜)
            ↕
    상대 AI 에이전트 ↔ 상대 사람
```

| 구분 | 기존 데이팅 앱 | TerminalLove |
|---|---|---|
| 사용 환경 | 모바일 앱 | 개발 에이전트 · 터미널 · IDE |
| 프로필 생성 | 사용자가 직접 작성 | **AI가 로컬에서 분석 후 생성** |
| 매칭 기준 | 사진·위치·관심사 | **성향·사고방식·대화 스타일** |
| 매칭 계산 위치 | 중앙 서버 | **로컬 (사용자가 튜닝 가능)** |
| 대화 지원 | 거의 없음 | **각자의 AI가 코칭** |
| 신원 | 플랫폼 계정 | **public key (서명으로 증명)** |
| 데이터 | 중앙에 집중 | **로컬 우선, 승인분만 공개** |
| 비교 대상 | Tinder/Bumble/Hinge | **Nostr / Matrix / MCP / DID** |

## 4. 포지셔닝 & 네이밍

- **카테고리에서 탈출**: "또 하나의 데이팅 앱"이 아니라 **"에이전트 시대의 새로운 소셜 레이어"**. Tinder와 유저풀·성비·사진 검증으로 싸우지 않는다.
- **개발자 명분**: "당신의 데이터는 로컬에. 당신의 에이전트가 당신을 분석. 프로필은 승인분만. 메시지는 암호화. 신원은 public key. 디렉토리/릴레이는 누구나 운영." → 개발자가 좋아할 문법.
- **완전 오픈소스 · 비영리**: 별도 수익화 모델 없음. hosted directory/relay는 편의를 위한 무료·커뮤니티 인프라이며, **유료 기능·연락처 과금은 두지 않는다.**
- **네이밍 (확정)**: `TerminalLove`로 통일
  - **`TerminalLove`** = 제품 · 프로토콜 · 레퍼런스 MCP 클라이언트 통합 명칭
  - 선택적 공식 hosted directory/relay = **무료·커뮤니티 인프라** (별도 유료 브랜드 없음)
  - `Agent Dating Protocol` = 영문 카테고리 설명어로만 사용(서브타이틀)
  - (검토했던 후보: CommitMate, MergeDate, DevCrush, AgentMatch, OpenMatch MCP, Pair Protocol)

## 5. 타깃 사용자 & 모드

**1차 타깃**: Claude Code / Codex / Cursor / Windsurf 등 AI 개발 도구 헤비유저, 사이드 프로젝트·글로벌 커뮤니티에 익숙한 개발자.
**2차 타깃**: 디자이너 · PM · 창업자 · 인디 해커 · 오픈소스 기여자 · 원격 근무자.

**매칭 모드 (모두 opt-in, 디렉토리/필터로 분리)**
- `Friend Mode` — 개발자 친구
- `Builder Mode` — 사이드 프로젝트 파트너
- `Founder Mode` — 공동창업자/메이커 네트워킹
- `Dating Mode` — 소개팅/연애 *(바이럴은 가장 강하지만 안전·성비 부담도 가장 큼 → opt-in 기본)*

**진입 전략**: 초기 유저 확보는 Friend/Builder가 쉽고, 바이럴은 Dating이 강하다. → **Builder/Friend로 부트스트랩 → Dating으로 바이럴 → 수익화 실험** 순서.

## 6. 시스템 아키텍처 (확정안)

### 6.1 전체 구조
```
[사용자 A]                                   [사용자 B]
Claude Code / Codex / Cursor                Claude Code / Codex / Cursor
        │                                           │
   TerminalLove MCP (self-hosted)           TerminalLove MCP (self-hosted)
   - 로컬 성향 분석                          - 로컬 성향 분석
   - signed profile 생성                     - signed profile 생성
   - 로컬 매칭 계산                          - 로컬 매칭 계산
   - E2E 암/복호화, 서명검증                 - E2E 암/복호화, 서명검증
   - 코칭 / sanitize                         - 코칭 / sanitize
        │                                           │
        └──────────────┬───────────────────────────┘
                       ▼
        ┌──────────────────────────────────┐
        │  선택적 Hosted Layer (필수 아님)  │
        │  - Public Directory (signed cards)│
        │  - Relay (암호문만 중계, 내용 X)  │
        │  - Push notification bridge        │
        │  - Verified community / proofs    │
        └──────────────────────────────────┘
   ※ 공식 directory/relay = 기본값. 커뮤니티/개인 directory·relay도 누구나 운영 가능.
   ※ 핵심 원칙: "No central platform required." (≠ "No servers at all.")
```

### 6.2 5개 레이어
| 레이어 | 책임 | 위치 |
|---|---|---|
| **1. Identity** | 로컬 키페어 생성, `agent_id` 발급, 서명/검증, 키 백업·교체 | 로컬 MCP |
| **2. Profile** | 성향 분석 → 서명된 공개 프로필 카드(`expires_at` 포함), 사용자 승인 | 로컬 MCP |
| **3. Discovery** | 디렉토리 게시/구독, 프로필 import/export, 초대 링크, **로컬 매칭 계산** | 로컬 MCP + 디렉토리(선택) |
| **4. Messaging** | intro → accept → E2E 암호화 메시지, replay 방지, relay 중계 | 로컬 MCP + relay(선택) |
| **5. Agent UX** | 터미널/IDE 안의 scan·알림·코칭·**1:1 대화** — **제품의 감성** | 로컬 MCP + 에이전트 |

> 이 제품의 핵심은 메시징 프로토콜이 아니라 **Agent UX + profile schema + local matching**이다. 메시징은 최소 스펙으로 충분.
>
> **컨텍스트 방화벽 (필수)**: 5번 레이어는 호스트의 **코딩 컨텍스트와 격리**되어 동작한다 — 코칭 LLM은 별도 컨텍스트, 알림·대화는 out-of-band 표면. 소개팅 내용은 진행 중인 작업 대화에 **절대 주입되지 않는다**(→ §10.4).

## 7. 데이터 모델 & 스키마 (구현용)

### 7.1 ID 체계
| ID | 의미 | 용도 |
|---|---|---|
| `agent_id` | 사람/에이전트 공개 신원 (= public key fingerprint 또는 `did:key`) | 발견·intro |
| `profile_id` | 공개 프로필 카드 | 디스커버리 |
| `intro_id` | 소개 요청 | accept/decline |
| `chat_id` | 실제 대화방 | send/reply |
| `msg_id` | 특정 메시지 | 메시지 단위 답장 |
| `alias` | (선택) 현재 상대에 붙이는 별명(`@sofia`) | 표시용. 1:1이라 보통 생략 |

**규칙**: 발견·intro는 `agent_id`. **활성 대화는 항상 0개 또는 1개(1:1)** — accept 시 단일 `chat_id`가 생기고, 이후 `send`/`reply`/`coach`/`open`은 **인자 없이 현재 대화를 자동 타깃**(ID 생략 가능).

### 7.2 Identity
```
private_key : 로컬에만 저장 (OS keychain 또는 보호된 파일)
public_key  : 공개
agent_id    : did:key:z6Mk... 또는 fingerprint(예: agent_8f3a91c2)
서명 알고리즘: Ed25519 (메시지·프로필 공통)
```

### 7.3 Signed Profile Card
```json
{
  "type": "profile_card",
  "version": "0.1",
  "owner": "agent_8f3a91c2",
  "display_name": "optional",
  "country": "Korea",
  "languages": ["Korean", "English"],
  "stacks": ["TypeScript", "Rust", "AI Agents"],
  "interests": ["Startups", "AI Products", "Side Projects"],
  "communication_style": "direct, logical, long-form",
  "matching_modes": ["dating", "builder", "friend"],
  "profile_confidence": 0.78,
  "expires_at": "2026-06-07T00:00:00Z",
  "proofs": [
    { "type": "github", "username": "abc", "verified_at": "..." },
    { "type": "domain",  "domain": "abc.dev", "verified_at": "..." }
  ],
  "signature": "..."
}
```
- **`expires_at` 기본 7일** (연장·revoke 가능). "내 소개팅 프로필이 인터넷에 영원히 남는" 부담을 줄임.
- 완전 삭제는 보장 불가(이미 받아간 복사본). → UX에서 **솔직히 고지**.

### 7.4 Encrypted Message
```json
{
  "type": "encrypted_message",
  "version": "0.1",
  "id": "msg_01",
  "from": "agent_8f3a91c2",
  "to": "agent_b7a2219d",
  "created_at": "2026-05-31T12:00:00Z",
  "conversation_id": "chat_82fa",
  "ciphertext": "...",
  "signature": "..."
}
```
- recipient public key로 암호화(X25519/NaCl box 계열), sender private key로 서명.
- relay가 보는 것: `from`, `to`, `timestamp`, `size`, 사용 패턴. **내용은 못 봄.**
- replay 방지: `id` + `created_at` + nonce.

## 8. 핵심 사용자 경험 (E2E 플로우)

```
1. 설치        claude mcp add terminallove   (또는 docker run terminallove/mcp)
2. /dating init        → 키페어 생성, agent_id 발급
3. /dating profile     → 로컬 성향 분석 → 공개 프로필 초안 → [수정][공개][취소]
4. /dating publish     → 선택 디렉토리에 signed card 게시
5. /dating scan        → 디렉토리에서 카드 수집 → 서명검증 → 로컬 매칭 → 추천
6. /dating intro <agent_id>  → 소개 요청 전송 (활성 대화가 없을 때만)
7. (상대) accept       → 단일 활성 chat 생성, encrypted chat 오픈
8. /dating open        → 현재 1:1 대화 + 코치 제안 → [edit/send/cancel]
9. /dating end         → 현재 대화 종료(언매치) → 그래야 다음 사람과 매칭 가능
   /dating block       → (선택) 현재 상대 일방향 차단
```

**터미널 UX 예시**
```
> /dating scan
Found 12 compatible profiles.

[1] agent_b7a2219d   Spain   TypeScript, Design Systems   87%
[2] agent_c91f02aa   Japan   Python, AI Agents            84%

> /dating intro agent_b7a2219d "Hi, I saw you're also building AI tools..."
Intro request sent. Waiting for acceptance.

🔔 Intro accepted. You're now connected 1:1.

> /dating open
@partner: 요즘 어떤 프로젝트 만들고 있어요?

Coach: 상대는 AI 제품·창작 도구에 관심이 많습니다. 기술보다 '왜 만드는지'를 함께 말하면 좋습니다.
Suggested reply: "요즘은 개발자용 AI 툴을 만들고 있어요. 평소 쓰는 개발 환경 안에서 자연스럽게..."
Send? [edit / send / cancel]

# 정리하고 다른 사람과 매칭하려면:
> /dating end           # 현재 대화 종료(언매치)
> /dating block         # (선택) 현재 상대 일방향 차단
```

## 9. 명령어 명세 (CLI 스펙)

| 명령어 | 설명 |
|---|---|
| `/dating init` | 키페어 생성, `agent_id` 발급 |
| `/dating whoami` | 내 `agent_id`/프로필 요약 |
| `/dating profile` | 성향 분석 → 공개 프로필 초안 생성/수정 |
| `/dating publish [--directory <name>]` | 서명 프로필 게시 |
| `/dating scan` | 디렉토리 수집 → 로컬 매칭 → 추천 목록 |
| `/dating profile <agent_id>` | 특정 후보 상세 보기 |
| `/dating intro <agent_id> ["first message"]` | 소개 요청(첫 메시지 선택). 활성 대화/대기 intro 있으면 차단 |
| `/dating cancel` | 보낸(대기 중) intro 취소 |
| `/dating inbox` | 받은 intro / 메시지 목록 |
| `/dating accept <intro_id>` · `/dating decline <intro_id>` | intro 수락/거절 (수락 시 **단일 활성 대화** 생성) |
| `/dating open` | 현재 1:1 대화 열기 (대화·코치 표시) |
| `/dating send "msg"` | 현재 대화에 메시지 전송 |
| `/dating reply [msg_id]` | 코치 제안과 함께 답장(특정 메시지 지정은 선택) |
| `/dating coach "draft"` | 작성한 초안 코칭/보정 |
| `/dating end` | **현재 대화 종료(언매치)** → 새 매칭 가능. 종료 상대는 재추천 제외 |
| `/dating block [agent_id]` | **일방향 차단**(기본=현재 상대). 재intro/메시지 불가, 상호 비노출 |
| `/dating report <agent_id> [reason]` | 신고 (커뮤니티 blocklist 반영) |
| `/dating alias <name>` | (선택) 현재 상대에 표시용 별명 부여 |
| `/dating export-profile` · `/dating import-profile <file>` | 프로필 내보내기/가져오기 |
| `/dating invite` | 초대 링크/QR 생성 |
| `/dating backup-key` · `/dating import-key` · `/dating rotate-key` | 키 관리 |

### 9.1 1:1 단일 대화 규칙 (핵심)
- **활성 대화는 항상 0개 또는 1개.** 새 사람에게 `intro` 하려면 현재 대화를 먼저 `/dating end`로 종료해야 한다. → 구현·관리·안전(스팸/무차별 메시지)이 모두 단순해짐.
- `agent_id`로는 **intro만** 가능. **DM 금지, intro-first**: 디렉토리에 프로필이 있다고 바로 메시지 불가. `발견 → intro → 상대 accept → 1:1 대화`.
- accept 후엔 `send`/`reply`/`coach`/`open`이 **인자 없이 현재 대화를 자동 타깃**.
- 활성 대화가 있는데 `intro` 시도 → 막고 안내:
  ```
  이미 @partner와 1:1 대화 중입니다. 새 사람과 매칭하려면 먼저 종료하세요:
  /dating end      (원하면 /dating block 으로 차단)
  ```

### 9.2 종료·차단 (대화가 끊겼을 때)
- **`/dating end` (언매치)**: 현재 1:1을 닫고 다시 매칭 가능 상태로. 종료한 상대는 로컬 **재추천 제외** 목록에 들어가 `scan`에 다시 안 뜸(영구 차단은 아님).
- **`/dating block` (일방향 차단, 기본)**: 차단 상대는 나에게 재intro·메시지 불가, 양쪽 `scan`에서 상호 비노출. 차단 사실은 상대에게 노출하지 않음(조용한 차단).
- **Cold 대화 자동 처리**: 기본 7일 무응답이면 자동 보관하고 `[종료 / 차단 / 계속]` 선택을 제안.
- **커뮤니티 blocklist (opt-in)**: 반복 가해자는 신고가 누적되면 구독형 공유 차단 리스트에 반영. **강제 양방향 blacklist는 두지 않음** — 한쪽이 대화를 끊었다는 이유로 상대를 전역 처벌하면 오남용 위험이 크다. 보호가 필요하면 `block`/`report`로 처리.

## 10. 보안 설계

### 10.1 신원·사칭 방지
- **클라이언트가 주장하는 sender를 믿지 않는다.** 메시지는 항상 sender private key로 **서명**, 수신 측이 public key로 **검증**. 검증 실패 → 폐기.
- (hosted relay 사용 시) relay/서버도 `from`을 신뢰하지 않고 토큰/서명 기반으로 판단. `match_id`/`chat_id` 기반 라우팅만 허용.

### 10.2 프롬프트 인젝션 방어 (필수)
상대 메시지가 내 에이전트에 전달되므로 **상대 메시지 = untrusted input.** 절대 system/developer context에 넣지 않는다.
```
수신 → MCP sanitize → 위험 패턴 검사 → LLM엔 'untrusted user message'로만 전달
     → 도구 실행/파일 접근 권한 차단 → 사용자에게 표시
```
**소개팅 컨텍스트에서 기본 차단할 권한**: 파일 읽기 · 터미널 실행 · 환경변수/API 키 접근 · 코드베이스 검색 · 외부 URL 자동 접속 · 계정 정보 접근.

### 10.3 Verified Client (해시는 강제 아닌 배지)
오픈소스라 포크가 자연스럽다. 공식 해시와 다르다고 무조건 차단하지 않는다.
- 표시: `Official MCP v0.1.4` / `Modified MCP (unverified)` / `Unknown`.
- **진짜 검증은 해시가 아니라 "이 메시지가 그 public key 소유자가 보낸 것인가"(서명).**

### 10.4 컨텍스트 방화벽 — 작업 ↔ 소개팅 완전 격리 (필수·핵심)

TerminalLove의 **어떤 활동도**(수신 메시지 · 발신 메시지 · 매칭 알림 · LLM 코칭) **진행 중인 코딩 세션의 LLM 컨텍스트(토큰 윈도우·대화 transcript)에 절대 들어가지 않는다.** 격리는 **양방향**이다.

| 방향 | 막는 것 | 이유 |
|---|---|---|
| **작업 → 소개팅** (§10.2·§11) | 코드·시크릿·작업 대화가 소개팅으로 새지 않음 | 프라이버시·인젝션 |
| **소개팅 → 작업** (신규·필수) | 소개팅 메시지/코칭/알림이 코딩 대화에 섞이지 않음 | 컨텍스트 오염·토큰 낭비·집중 방해·transcript 노출 방지 |

**구현 원칙**
- **별도 컨텍스트의 코칭 LLM**: 대화 코칭은 호스트 에이전트의 메인 대화가 아니라 **소개팅 데이터만 시드된 독립 LLM 호출/서브세션**에서 수행한다. 코딩 컨텍스트와 메모리·히스토리를 공유하지 않는다.
- **사이드카 데몬**: TerminalLove는 백그라운드 데몬으로 동작하며, 메시지/코칭 결과를 메인 대화의 tool result로 주입하지 않는다.
- **Out-of-band 표면**: 매칭/메시지/대화는 메인 transcript가 아니라 **별도 채널**(전용 TUI·페인, 상태줄 배지, OS/터미널 알림, 별도 `/dating` REPL)에서 표시·입력한다.
- **MCP 한계 주의**: 일반 MCP tool 호출의 결과는 호스트의 메인 컨텍스트로 들어간다. 따라서 소개팅 본문·코칭은 **메인 대화에 노출되는 tool 출력으로 흘리지 않고** 별도 표면으로 라우팅한다. (메인에는 "🔔 새 메시지 1건" 같은 **알림 한 줄만** out-of-band로 띄우는 정도는 허용 — 단, 본문/코칭은 절대 포함하지 않음.)
- **명시적 진입·복귀 차단**: 사용자가 소개팅 화면을 열기 전엔 코딩 세션에 아무것도 안 들어오고, 열어서 본 내용도 코딩 컨텍스트로 되돌아가지 않는다.

> 즉, 코딩 에이전트는 "내가 누군가와 소개팅 대화 중"이라는 사실조차 모른다. 같은 머신에서 돌지만 **두 컨텍스트는 물리적으로 분리**된다.

### 10.4.1 구현 메커니즘 — 클라이언트별 (Claude Code / Codex, 2026-06 확인)

**검증된 핵심 사실**: Claude Code·Codex 모두 ⓐ **MCP tool 결과는 항상 메인 컨텍스트로 주입**되고, ⓑ **subagent는 격리 컨텍스트지만 최종 요약이 메인으로 복귀**한다. → 따라서 **대화·코칭을 메인 세션의 MCP tool이나 subagent로 노출하면 안 된다.** 완전 격리의 유일한 확실한 보장은 **"별도 프로세스/세션"** 이며, 이는 타협이 아니라 정석이다. 아래 out-of-band 기능들이 이를 매끄럽게 만들어 준다.

**공통 3-파트 구조**
1. **백그라운드 데몬** — relay/인박스를 watch, 로컬 DB 보관. 메인 대화와 무관하게 동작.
2. **Out-of-band 알림 브리지** — "🔔 새 메시지 1건"만 컨텍스트 밖으로 표시(본문·코칭은 절대 포함 안 함).
3. **별도 세션/표면** — 실제 대화 + 코칭 LLM은 독립 프로세스에서. 메인 코딩 컨텍스트·transcript와 완전 분리.

| 항목 | Claude Code | Codex |
|---|---|---|
| 알림 브리지 | **Hook**(exit 0·stdout 없음 = 컨텍스트 미주입). `Notification`/`SessionStart` 등 | **`notify`**(`~/.codex/config.toml`, `agent-turn-complete`에 외부 프로그램 실행) + **hooks**(`~/.codex/hooks.json`, 출력은 transcript 미포함) |
| 경량 상태 표시 | **statusLine**(커맨드 기반, 모델 비가시) → "🔔 N건" 표시 가능 ✓ | status line은 **고정 항목만** 지원, 커스텀 텍스트 미배포(Issue #20244) ✗ → 터미널 ping/`notify`로 대체 |
| 격리 세션 | 별도 `claude` 세션/페인(독립 transcript). subagent는 요약이 메인 복귀라 부적합 | 별도 `codex` 세션 또는 **`codex exec` + 전용 `--profile`**(자체 `mcp_servers`). subagent도 요약 복귀라 부적합 |
| 메시징 MCP 등록 | 메인 세션에 **등록하지 않음** | 메인 `mcp_servers`에 넣지 않고 **전용 profile/세션에만** 등록 |

**원칙**: 메인 코딩 세션에는 TerminalLove를 대화/코칭 용도로 붙이지 않는다. 메인에는 **알림 한 줄(hook/notify/statusLine)만** 노출하고, 사용자가 그걸 보고 **별도 세션/창을 열어** 대화한다. 그 별도 세션이 데몬 DB를 읽고 코칭 LLM(독립 컨텍스트)을 구동한다.

> 주의: Codex `notify`는 `agent-turn-complete` 1개 이벤트에만 발화하며 프로젝트 로컬 config에선 무시됨(유저 레벨 `~/.codex/config.toml` 필요, payload 키는 kebab-case). Codex statusline 커스텀 텍스트는 아직 미배포 → Codex 경량 표시는 터미널 ping/`notify`로 처리.

### 10.4.2 알림 정책 (확정)

메인 *화면*에 **알림 한 줄 · 알림음 · 데스크톱 토스트는 허용**한다. 단, 이는 전부 **UI 레이어**(statusLine / 사운드 / OS 알림)로만 노출하며 **모델이 읽는 대화 컨텍스트(토큰·transcript)에는 본문은 물론 알림 라인조차 주입하지 않는다.** ("화면에 보임" ≠ "컨텍스트에 들어감".)

- **트리거 = 데몬**: 메시지 도착 사운드/토스트는 relay를 watch하는 **데몬**이 직접 발생시킨다. (Claude Code/Codex의 hook은 *에이전트 자체 이벤트*에만 발화하고 *외부 메시지 도착*은 감지하지 못하므로 도착 알림은 데몬 담당. hook은 "세션 시작 시 대기 N건 표시" 같은 보조로만.)
- **지속 표시**: Claude Code는 `statusLine`에 "🔔 N건"(데몬 상태파일 읽기). Codex는 statusline 커스텀 미지원 → 터미널 벨/`notify` 사운드로 대체.
- **본문·코칭**: 오직 별도 세션(대화 화면)에서만 표시. 코딩 세션에는 절대 미노출.
- **알림 라인 내용 한도**: 카운트/발신 alias 정도까지만(예: "🔔 @sofia 새 메시지"), 메시지 본문·코칭은 금지.

## 11. 프라이버시 설계

**원칙 3종**: `Local-first profile generation` · `User-approved public profile` · `Central/relay as router, not memory`.

| 올라가는 것 (승인분) | 절대 올리지 않는 것 |
|---|---|
| 공개 성향 요약, 매칭 선호 | Claude/Codex 원문 대화 |
| 국가/언어/스택/관심사 | 로컬 코드베이스·메모·비공개 작업 |
| 검증 상태, 라우팅 메타 | API 키·토큰·환경변수 |
| 신고/차단 상태(hosted) | 개인 연락처, 미승인 세부 분석 |

- 프로필 `expires_at`(기본 7일)·revoke 지원.
- **Metadata leakage 솔직 고지**: E2E라도 relay는 from/to/timestamp/size를 본다. 문서에 명시.

## 12. 신뢰 & 안전 (Trust & Safety)

> **cryptographic identity ≠ "좋은 사람" 증명.** 서명은 출처만 증명한다. 안전·매너·실재·신고이력은 별도 신뢰 레이어가 필요.

**신뢰 신호(중앙 권위 아님, 선택형)**: GitHub proof · domain proof · community membership proof · local blocklist · community blocklist · relay-level moderation · optional verified directory · invite graph.

**안전 기본 정책 (성비·괴롭힘 방어, 중앙 검열 없이 UX로)**
- intro request 하루 제한, 같은 사람 재요청 제한
- 상대가 accept해야 채팅 시작 (intro-first)
- 첫 메시지 길이 제한
- local blocklist 기본 제공 + community blocklist 구독(opt-in)
- 대화 종료(`/dating end`) · 일방향 차단(`/dating block`) · 신고(`/dating report`) 기본 제공 — 강제 양방향 blacklist는 미도입(§9.2)
- directory별 moderation policy 공개
- **Dating Mode는 opt-in**, 여성 유저 보호용 rate limit·reputation 반영

**연락처 공유**: 중앙에서 막지 않되 **항상 사용자 명시 승인 필요, 자동 추출/자동 전송 금지.**
```
개인 연락처가 포함되어 있습니다: +82-10-xxxx-xxxx
정말 전송할까요? [취소][전송]

상대가 이메일을 공유했습니다: example@gmail.com
[무시][저장][답장에 사용]
```

> MVP 성공 기준 = **"Claude Code 쓰다가 소개팅 매칭 알림이 온다"가 실제로 작동하고 공유된다.**

## 16. 기술 스택 권장안 (구현 기준)

| 영역 | 권장 | 비고 |
|---|---|---|
| MCP 서버 | **TypeScript + 공식 MCP SDK** | Claude Code·Codex MCP 생태계가 TS 우선. Python도 가능 |
| 우선 타깃 클라 | **Claude Code + Codex** | 두 클라이언트 동시 타깃. Cursor/Windsurf는 이후 |
| 신원/서명 | **Ed25519** (libsodium/`@noble/ed25519`) | `did:key` 호환 |
| E2E 암호화 | X25519 + NaCl box(`@noble/...` 또는 libsodium) | recipient 공개키 암호화 |
| 로컬 저장 | SQLite 또는 파일 + OS keychain(키) | 로컬 우선 |
| Directory/Relay | reference 서버(경량 HTTP/WS) | 누구나 self-host 가능하게 |
| 배포 | `npm` 패키지 + `docker run terminallove/mcp` | 한 줄 설치 |
| 알림/대화 표면 | **out-of-band**(상태줄·별도 페인·OS 알림·별도 `/dating` REPL) | **메인 코딩 컨텍스트에 미주입** — 컨텍스트 방화벽(§10.4) |
| 코칭 LLM | **별도 컨텍스트의 독립 호출** | 코딩 세션과 분리 |

> 프로토콜 자체는 **자체 minimal spec**으로 시작하되 Nostr/Matrix/DID에서 구조를 차용(아래 §20).

## 17. 핵심 지표 & 가설

**지표** (가입자 수보다 매칭·대화 품질):
MCP 설치 수 · 공개 프로필 승인율 · 첫 매칭 수락률 · **첫 메시지 전송률** · 24h 응답률 · **3일 이상 대화 지속률** · 신고/차단율 · 연락처 공유 요청/성공률.
→ **북극성 3종: ① 프로필 승인율 ② 첫 메시지 전송률 ③ 3일 이상 지속률.**

**검증 가설**:
1. 개발자는 별도 앱보다 에이전트 안 매칭을 더 신박하게 느낀다.
2. AI 생성 성향 카드가 직접 작성 프로필보다 흥미·신뢰가 높다.
3. AI 대화 코칭이 실제 대화 지속률을 높인다.
4. self-hosted 구조가 프라이버시 신뢰를 준다.
5. Friend/Builder/Founder 모드 병행이 초기 유저 확보를 쉽게 한다.

## 19. 로드맵

- **Phase 1 — 바이럴 프로토타입**: Claude Code·Codex MCP, 로컬 성향 카드, 제한적 매칭, out-of-band 알림, 답장 추천. 성공기준: 커뮤니티 공유 + "나오면 써보고 싶다" 반응.

- **Phase 3 — Open Beta**: GitHub 로그인(선택), connection 관리, 모드 분리, 글로벌 매칭/번역, 커뮤니티 디렉토리 확장.
- **Phase 4 — Protocol화**: Open-source MCP SDK, profile schema 공개, Matching API, Verified Client Registry, 서드파티 클라이언트/플러그인.

## 20. 기술 선택지 비교 & 결론

| 선택 | 장점 | 단점 |
|---|---|---|
| **A. 자체 minimal spec** | dating/agent UX 최적화, 브랜드 독립 | relay/directory 생태계를 처음부터, 보안검증 부담 |
| **B. Nostr 위 dating event kind** | public key+relay 재사용, 빠른 구현 | Nostr 이미지에 끌려감, dating UX 우회 필요 |
| **C. Matrix federation** | 메시징/멀티디바이스 성숙 | MVP엔 무거움, agent UX 래핑 필요 |

**결론**: **A(자체 minimal spec)로 시작하되 Nostr(keypair/signed event/relay) · Matrix(federation/E2E) · DID(did:key)에서 구조를 적극 차용.** 제품 핵심은 메시징이 아니라 *분석 → 프로필 → 로컬 매칭 → 코칭*이므로 메시징 프로토콜에 종속되지 않는다.

## 21. 확정된 결정 (2026-06-01)

1. **네이밍**: `TerminalLove`로 통일. ("Agent Dating Protocol"은 카테고리 설명어.)
2. **언어/런타임**: TypeScript + 공식 MCP SDK. 암호 `tweetnacl`(Ed25519 서명 / X25519 box).
3. **우선 클라이언트**: **Claude Code + Codex** (두 개 동시 타깃).
4. **수익화**: **없음 — 완전 오픈소스.** hosted directory/relay는 무료·커뮤니티 인프라.
5. **대화 모델**: **1:1 전용, 동시 활성 대화 1개.** 새 매칭 전 `/dating end` 필요.
6. **안전**: 일방향 `/dating block`(기본) + `/dating end`(언매치) + `/dating report`. 강제 양방향 blacklist 미도입, 커뮤니티 blocklist는 opt-in.
7. **컨텍스트 방화벽**: 소개팅 메시지·코칭·알림은 진행 중인 코딩 세션 컨텍스트에 **절대 주입 안 함** (별도 컨텍스트 코칭 LLM + out-of-band 표면). → §10.4.
8. **Phase 1 범위**: **단일 머신 데모 풀 플로우** — 두 identity(Alice/Bob) 시뮬레이션으로 `init→profile→publish→scan→intro→accept→open/send→coach (+end/block)`. 디스커버리=로컬 공유 디렉토리, relay=로컬 공유 인박스.
